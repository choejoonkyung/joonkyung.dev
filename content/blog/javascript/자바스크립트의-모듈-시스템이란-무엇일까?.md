최근 자바스크립트를 알게 되었다면 **모듈 시스템**이라는 단어가 생소할 것 같다. 다른 언어들은 코드들을 모듈화하고 가져오는 것(import)은 기본적으로 지원하기 때문에 “이 기능을 따로 구분 지어 부르긴 하는가?” 의문이 들기도 합니다. 자바스크립트에서 모듈 시스템은 무엇일까요.

## 자바스크립트를 브라우저 밖에서도 사용하자

> javascript: not just for browsers any more!

자바스크립트의 모듈 시스템을 이해하려면 그동안의 자바스크립트의 변천사를 이해한다면 조금이나마 와닿기 시작합니다. 자바스크립트는 동적으로 변하는 콘텐츠와 웹페이지의 외관을 만들기 위해 등장하였습니다. 이때까지만 해도 자바스크립트는 브라우저 안에서만 동작했습니다. 그러면서 AJAX의 등장으로 인하여 웹 사용성이 많이 개선되었고 자바스크립트의 코드량이 증가하면서 많은 연산이 필요한 상황이 이어졌습니다. 이러한 흐름 속에 V8 엔진이 등장하면서 기존의 자바스크립트 엔진보다 월등히 빨라졌습니다. 그래서인지 자바스크립트를 브라우저 용만이 아닌 서버사이드로 빼고 싶은 사람들이 조금씩 나오기 시작했습니다.

이부터는 자바스크립트를 브라우저 밖에서도 동작하자는 뜻을 가진 사람들이 모였고 이 그룹은 **자바스크립트가 브라우저 밖에서 동작하면서 모두가 범용적으로 사용하고자한다면 표준을 정하고 지켜야 된다**고 생각했습니다. 이렇게 시작한 그룹의 이름, 프로젝트 이름이 바로 **Common JS**입니다.

## 서버사이드 자바스크립트의 쟁점

이전 브라우저용 자바스크립트에서는 모듈에 대한 정의가 없었습니다. 단지 `<script>` 태그로 가져와 사용하였습니다. 그러다보니 따로 파일로 나눠서 태그로 가져와도 모두 글로벌 변수로 생성되어 겹치는 경우가 생기게 되었습니다. 또한 모듈에 대한 정의가 없다보니 내 코드를 패키징해서 배포할 수 없었고 다른 사람 또한 다른 사람이 배포한 모듈을 가져올 수 없었습니다.

이후 여러가지 문제들이 많았고 결국 자바스크립트가 브라우저용에서 벗어나 범용적으로 쓰일려면 어떤 모습이여야 하는가에 대한 고민은 결국 하나로 귀결되었습니다. 바로 **모듈화**입니다.

## CommonJS 모듈화 명세

CommonJS의 모듈화는 아래와 같이 세 부분으로 이루어졌습니다.

- 스코프(Scope): 모든 모듈은 자신만의 독립적인 실행 영역이 있어야 한다.
- 정의(Definition): 모듈 정의는 exports 객체를 이용한다.
- 사용(Usage): 모듈 사용은 require 함수를 이용한다.

모듈은 자신만의 독립적인 환경이 있어야됩니다. 모듈마다 독립적인 환경이기 때문에 모듈에서 또 다른 모듈을 가져와 사용이 가능합니다. 이는 글로별 변수로 생성되던 이전 문제를 해결해주며 코드를 삽입하는 방식을 개선하여 다른 사람의 코드를 가져올 수 있기에 패키지 배포/삽입 문제도 해결됩니다. 독립적인 환경은 어떻게 구성할까? 자바스크립트는 파일 마다 스코프가 존재합니다. CommonJS는 파일 하나에 모듈 하나를 작성하여 해결하였습니다.

### 모듈화

```jsx
// math.js
module.exports = {
  sum: function(a, b) {
    return a + b
  },
}

// main.js
const sum = require('./math.js').sum
console.log(sum(10, 90)) // 100
```

math.js 파일의 exports에 sum 함수을 담아 공유하게 됩니다. 이는 main.js 파일에서 모듈을 불러와 sum 함수를 사용할 수 있게 됩니다. 방식 굉장히 단순하고 직관적이라 사용하기 편하다는 점이 있습니다.

하지만 앞써 언급한대로 CommonJS는 자바스크립트가 브라우저밖에서 동작해보자라는 의도에서 출발했습니다. 그래서인지 브라우저 관점에서 모듈화는 미흡한 부분이 있습니다. 바로 파일을 가져와야 하는 상황입니다. 서버사이드에서는 파일을 바로 가져오는데에 문제가 없었지만 브라우저에서 파일을 가져오는 상황에서는 파일을 통신을 통해 받아와야하기 때문에 동기적으로 처리해야 됩니다. CommonJS는 이런 부분이 미흡했습니다.

결국 CommonJS 그룹 내에서도 브라우저 환경도 신경쓰고 싶은 그룹이 따로 독립하게 되었습니다. 그 그룹이 **AMD**(Asynchronous Module Definition)입니다.

## ES Module (ECMAScript가 제시한 표준)

그럼 서버사이드. 클라이언트사이드 모두 다른 모듈 시스템을 사용해야할까요? 이런 문제는 RequieJs등 모듈 로더들이 많이 등장하여 사용되어 왔습니다.

하지만 이제는 표준화된 구문을 가지고 있습니다. 바로 **ESM**입니다. 모듈 내에서 export 키워드를 사용하여 거의 모든 항목을 내보낼 수 있습니다. 상수, 함수 또는 다른 변수 바인딩 또는 선언을 내보낼 수 있습니다. 변수 문이나 선언 앞에 내보내기만 하면 됩니다.

### 모듈화

```jsx
// 📁 lib.mjs
export default function(string) {
  return `${string.toUpperCase()}!`
}

// 📁 main.mjs
import shout from './lib.mjs'
//     ^^^^^
```

최근 JS를 접하셨다보면 가장 흔하게 볼 수 있었던 모듈화 방식입니다.

## 마치면서

내가 처음 자바스크립트을 접했을 때에는 이미 ESM으로 자리 잡은 상태여서 노드에서 서버를 개발할 때에도 React로 클라이언트를 개발할 때에도 동일한 모듈 시스템을 사용했다. 그래서인지 왜 모듈 시스템이라고 따로 불리는지, 여러가지인지 이해가 되지 않았다. 자바스크립트의 변화을 바라보면서 와닿기 시작했고 이해하게 되었다.
